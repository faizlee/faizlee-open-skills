# 设计决策记录

## 不使用 `--last`

原因：

- 多 pair 并行时 `--last` 容易恢复到错误会话。
- pair 必须绑定明确 `codexSessionId`，保证 Codex 裁决来自指定指挥线程。

## 不使用 subagent

原因：

- 目标是用户级轻量 relay，不是 Codex 内部委派链。
- subagent 会增加上下文、额度和可追踪复杂度。
- pair 的执行工程师是外部 Claude Code 会话。

## 不启动 `codex-with-cc`

原因：

- 本工具只维护固定 pair 和文件协议。
- `codex-with-cc` 更像完整自动委派链，自动化和审计强，但额度和控制复杂度更高。
- 当前工具更适合用户掌控、按 pair 复盘和低成本协作。

## 不做 daemon

原因：

- 用户希望明确知道何时消耗 Codex / Claude Code。
- daemon 容易产生隐性循环和后台额度消耗。
- 目前用 Dashboard server + 一次性 runner，用户关闭窗口即可停控制器。

## 不做窗口注入

原因：

- 自动向另一个终端输入命令不可审计，容易误触。
- 不同终端、权限、焦点、编码差异大。
- 更稳定的做法是用 CLI 参数或文件状态推进。

## 为什么还支持前台终端

原因：

- 用户需要看到接近原生 TUI 的执行过程。
- 前台终端用于观察和人工接管，不用于隐藏式注入。
- 自动化主路径仍应靠 runner 文件和状态机。

## 为什么需要 runner

原因：

- Dashboard 是 HTTP 请求，不适合长时间同步阻塞。
- Runner 把长任务独立成进程，并把状态、stdout、stderr、结果写成文件。
- 这样页面可以刷新查看，也可以停止或诊断。

## 为什么要区分进程状态和业务状态

原因：

- 进程完成不等于任务完成。
- 例如 summary runner 只检查缓存也会 `completed`，但业务结果可能是 `cache miss`。
- Dashboard 必须显示两层状态，避免用户误判。

## 为什么总结不覆盖

原因：

- CC、Codex、本地规则生成的总结视角不同。
- 应分别保存在 `summary/cc/`、`summary/codex/`、`summary/local/`。
- `latest` 可以指向各自分析方式的最新结果，但不要互相覆盖。
